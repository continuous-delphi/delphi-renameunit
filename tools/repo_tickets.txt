Title: Log after-line extracted at wrong offset in modified source
Issue: #3
Status: closed
Labels: bug
Body:
In `ProcessFile` (Engine.pas line 548), after applying replacements the "after" line
is extracted from `Modified` using the original source offset:

```delphi
AfterLine := ExtractSourceLine(Modified, FReplacements[I].StartOffset);
```

`FReplacements[I].StartOffset` is a 0-based offset into the **original** source.
After replacements that change string length (which is nearly always the case),
this offset points to the wrong position in `Modified`, producing an incorrect
or garbage "after" line in the log.

Fix approach: extract the after-line by re-tokenizing, by tracking cumulative
offset shifts, or by extracting the line from `Modified` using line number
rather than byte offset.

Acceptance criteria:
- [ ] Log after-lines correctly reflect the replacement result
- [ ] Add a test that verifies log output for a rename that changes name length

Title: TStringList round-trip may alter .dproj line endings
Issue: #4
Status: closed
Labels: bug
Body:
`ProcessDprojFile` reads source into a string, then assigns it to `TStringList.Text`
(line 609), and later reads back via `Lines.Text` (line 660). This round-trip:

- Normalizes all line endings to CRLF (sLineBreak)
- May add a trailing line break if one was absent

For .dproj files that use LF-only endings or have no trailing newline, the output
will differ from the input beyond the intended DCCReference replacements. This
creates unnecessary diffs and could cause merge conflicts.

Fix approach: operate on the raw source string directly (find/replace the
`Include="..."` attribute value) rather than splitting into TStringList lines.

Acceptance criteria:
- [ ] .dproj files are not modified beyond the intended DCCReference replacements
- [ ] Line endings and trailing newline state are preserved

Title: ProcessFile adds UTF-8 BOM to .pas/.dpr/.dpk files
Issue: #5
Status: closed
Labels: bug
Body:
`ProcessFile` line 562 writes modified source files using:

```delphi
TFile.WriteAllText(FilePath, Source, TEncoding.UTF8);
```

`TEncoding.UTF8` includes the 3-byte BOM preamble. If the original file had no BOM
(common for .pas/.dpr/.dpk), the tool silently prepends one. This creates unwanted
diffs and may confuse tools that don't expect BOM in source files.

The .dproj path (ProcessDprojFile line 661) is fine since RAD Studio writes .dproj
with BOM.

Fix approach: use a BOM-less UTF-8 encoding for ProcessFile writes. For example,
create a `TUTF8EncodingNoBOM` or use `TEncoding.UTF8` with preamble stripped.
Alternatively, detect whether the original file had a BOM and preserve that state.

Acceptance criteria:
- [ ] .pas/.dpr/.dpk files are written without BOM (unless they originally had one)
- [ ] .dproj files continue to be written with BOM

Title: Unit declaration handler matches prefix of longer dotted unit name
Issue: #10
Labels: bug
Status: closed
Body:
The unit declaration handler in `FindReplacements` (Engine.pas line 339) does
not check `FollowedByDot` after matching a token sequence. When processing a
map file with sequential pairs, a shorter unit name matches as a prefix of a
longer one in the `unit` declaration, corrupting the longer name.

### Reproduction

Map file (pairs applied in order):
```
Delphi.Lexer=Prefixed.Delphi.Lexer
Delphi.Lexer.Scanner=Prefixed.Delphi.Lexer.Scanner
```

Source file `Delphi.Lexer.Scanner.pas`:
```pascal
unit Delphi.Lexer.Scanner;
```

After processing:
1. Pair `Delphi.Lexer` fires on the unit declaration -- matches
   `Delphi.Lexer` (2 parts), no FollowedByDot check, replaces it.
   Declaration becomes `unit Prefixed.Delphi.Lexer.Scanner;`
2. Pair `Delphi.Lexer.Scanner` runs next -- looks for `Delphi.Lexer.Scanner`
   but finds `Prefixed.Delphi.Lexer.Scanner` -- no match.
3. File is NOT renamed (unit declaration no longer matches the pair).

The uses clause handler (line 384-386) is NOT affected because it checks
`not FollowedByDot(Tokens, EndIdx)` and correctly rejects prefix matches.

### Root cause

Line 339:
```pascal
if MatchTokenSequence(Tokens, NextI, FFromParts, EndIdx) then
```

Missing: `and not FollowedByDot(Tokens, EndIdx)`.

In a `unit` declaration, the unit name is always complete -- a dot following
the matched sequence means the match is a prefix, not the full name.

### Fix

Add the FollowedByDot guard to the unit declaration match:

```pascal
if MatchTokenSequence(Tokens, NextI, FFromParts, EndIdx) and
   not FollowedByDot(Tokens, EndIdx) then
```

### Workaround

Sort map entries by name length descending (longest first) so longer names
are renamed before shorter prefixes can interfere. This is what
radFormatter's `sync-parser.ps1` does. The workaround is reliable but the
tool should be correct regardless of map entry order.

### Acceptance criteria

- [ ] Unit declaration handler rejects prefix matches via FollowedByDot
- [ ] Map entry order does not affect correctness for hierarchical names
- [ ] Test: `Delphi.Lexer` does not match inside `unit Delphi.Lexer.Scanner;`
- [ ] Test: `Delphi.Lexer.Scanner` still matches `unit Delphi.Lexer.Scanner;`
- [ ] Existing tests still pass
